Too Many Script Statements: 200001

Your Apex code executed more than 200,000 individual script statements in a single transaction, exceeding the Salesforce governor limit for total Apex statement executions.

error
System.LimitException: Too many script statements: 200001

What does this error mean?

Salesforce counts every Apex statement executed — assignments, method calls, loop iterations, conditionals — and caps the total at 200,000 per transaction. This limit is separate from the CPU time limit: code can hit 200k statements before the CPU timer fires if each statement is cheap but numerous.

Every loop iteration increments the counter. A triple-nested loop over three lists of 100 items each consumes 1,000,000 potential statement slots — five times the limit — before any business logic inside the body runs.

Common Causes

1. Nested loops over large collections

O(n²) or O(n³) algorithms — matching records from two or three large lists using nested for loops — multiply statement counts exponentially as data volume grows.

Visualforce + JS
/* ❌ BAD: recursive Apex called via JS Remoting with no guard */
Visualforce.remoting.Manager.invokeAction(
    'MyController.processAll',
    recordIds,
    function(result, event) { /* no error handling */ }
);

2. Recursive utility methods without depth guards

A recursive method traversing a deep object graph (e.g., a hierarchy of custom objects) or processing a large tree structure can fire thousands of recursive calls, each adding to the counter.

3. Processing all records in memory instead of in Batch Apex

Triggering a full-org data migration or mass update synchronously — rather than chunking the work through Batch Apex — means all iterations run in a single transaction against the 200k budget.

How to Fix It

Solution 1: Move large processing to Batch Apex

Each execute() invocation in a Batch Apex job starts a fresh transaction with its own 200,000 statement budget. Processing 200 records per batch rather than 200,000 at once effectively eliminates this limit.

Apex — Correct
// ✅ Move heavy iteration to Batch Apex
public class ProcessAllBatch implements Database.Batchable<SObject> {
    public Database.QueryLocator start(Database.BatchableContext bc) {
        return Database.getQueryLocator(
            'SELECT Id FROM Account WHERE Active__c = true'
        );
    }
    public void execute(Database.BatchableContext bc,
                         List<Account> scope) {
        // Each execute() gets its own 200k statement budget
        for (Account a : scope) { /* process */ }
    }
    public void finish(Database.BatchableContext bc) {}
}
Database.executeBatch(new ProcessAllBatch(), 200);

Solution 2: Replace nested loops with Map lookups

Every nested loop can be replaced with a pre-built Map, reducing complexity from O(n²) to O(n) and dramatically cutting statement count.

Apex — Optimise Loops
// ❌ BAD: statement-heavy nested loop
for (Account a : accounts) {
    for (Contact c : allContacts) {
        if (c.AccountId == a.Id) { /* many statements per pair */ }
    }
}

// ✅ GOOD: map lookup collapses O(n²) to O(n)
Map<Id,List<Contact>> byAccount = new Map<Id,List<Contact>>();
for (Contact c : allContacts) {
    if (!byAccount.containsKey(c.AccountId))
        byAccount.put(c.AccountId, new List<Contact>());
    byAccount.get(c.AccountId).add(c);
}
for (Account a : accounts) {
    List<Contact> contacts = byAccount.get(a.Id);
}
💡
Pro Tip: Use Limits.getScriptStatements() and Limits.getLimitScriptStatements() at key checkpoints in complex methods to measure consumption. Adding a debug line like System.debug(Limits.getScriptStatements() + ' statements used') before and after a suspect loop identifies exactly where your budget is being spent.
