DML Error INVALID_CROSS_REFERENCE_KEY

INVALID_CROSS_REFERENCE_KEY

A lookup or relationship field was assigned an Id that belongs to the wrong SObject type, does not exist in the org, or comes from a different org entirely.

error
INVALID_CROSS_REFERENCE_KEY: invalid cross reference id

What does this error mean?

Every Salesforce record Id encodes the SObject type in its first three characters (the key prefix). When you assign a value to a lookup or relationship field, the platform validates that the Id's key prefix matches the expected SObject type for that field. If the prefixes don't match — or if the Id doesn't exist in the current org — you get INVALID_CROSS_REFERENCE_KEY.

Anatomy of a Salesforce Id (18-char)
001 DB000000 000000AAA
↑ Key Prefix (SObject type) ↑ Org Id portion ↑ Record identifier
001 = Account, 003 = Contact, 006 = Opportunity, 00Q = Lead, 005 = User …

The error is a DML-time failure — it appears in the Database.SaveResult errors array when using partial DML, or throws an System.DmlException when using standard DML statements.

Common Causes

1. Assigning a Contact Id to an Account lookup (or vice versa)

A common data-mapping mistake when processing external records or migrating data: the wrong Id variable is assigned to the wrong field because both are declared as Id type and the compiler doesn't catch cross-object mismatches.

Apex — Bad Example 
// ❌ BAD: contactId assigned to AccountId field
Id contactId = '0030000000AbcDEF'; // 003 prefix = Contact
Opportunity opp = new Opportunity(
    Name = 'New Deal',
    StageName = 'Prospecting',
    CloseDate = Date.today().addDays(30),
    AccountId = contactId // 💥 wrong type — expects 001 prefix
);
insert opp; // throws INVALID_CROSS_REFERENCE_KEY

2. Hard-coded Ids copied from sandbox that don't exist in production

Developer sandboxes have different record Ids than production. Hard-coding a User Id, RecordType Id, or any reference from a sandbox and deploying that code to production produces this error at runtime.

3. Cross-org Ids in integration payloads

An inbound integration payload contains a Salesforce Id from a different org (e.g., a customer's org sending their own record Ids). Those Ids are syntactically valid but referentially meaningless in your org.

How to Fix It

Solution 1: Verify the Id prefix before assignment

Use the Id.getSObjectType() method to assert that an Id belongs to the expected SObject type before assigning it to a relationship field. This catches mismatches at runtime with a clear, actionable error.

Apex — Correct 
// ✅ GOOD: validate Id type before assigning
public static void linkOpportunity(Opportunity opp, Id parentId) {
    Schema.SObjectType expectedType =
        Schema.Account.getSObjectType();

    if (parentId.getSObjectType() != expectedType) {
        throw new IllegalArgumentException(
            'Expected an Account Id but received a ' +
            parentId.getSObjectType().getDescribe().getName() +
            ' Id: ' + parentId
        );
    }
    opp.AccountId = parentId;
    insert opp;
}

Solution 2: Replace hard-coded Ids with queries

Never hard-code record Ids in Apex. Query for the record by a unique, stable field (Name, DeveloperName, or a custom external Id field) at runtime so the code works in all environments.

Apex 
// ❌ BAD: hard-coded Id works only in one org
Id rtId = '012000000000AbcAAE';

// ✅ GOOD: query by DeveloperName — works everywhere
Id rtId = Schema.SObjectType
    .Opportunity
    .getRecordTypeInfosByDeveloperName()
    .get('Enterprise')
    .getRecordTypeId();

// For User or other records:
User integrationUser = [
    SELECT Id FROM User
    WHERE Username = 'integration@mycompany.com'
    LIMIT 1
];

Solution 3: Validate external Ids in integration payloads

When processing inbound data from external systems, treat all Id fields as untrusted input. Use Database.query() to verify the record exists before referencing it in a DML operation.

Apex 
public static Id validateAccountId(String rawId) {
    try {
        Id safeId = (Id) rawId;
        if (safeId.getSObjectType() != Account.getSObjectType()) {
            return null;
        }
        List<Account> accs = [
            SELECT Id FROM Account WHERE Id = :safeId LIMIT 1
        ];
        return accs.isEmpty() ? null : accs[0].Id;
    } catch (Exception e) {
        return null; // malformed Id string
    }
}
💡
Pro Tip: The key prefix for every standard object is documented in Salesforce's Key Prefix Reference. For custom objects, run SELECT KeyPrefix FROM EntityDefinition WHERE QualifiedApiName = 'MyObject__c' in Workbench or Developer Console to identify the prefix. Bookmarking a full key prefix list helps you diagnose mismatches instantly when reading log files.
