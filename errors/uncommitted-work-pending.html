Runtime Error UNCOMMITTED_WORK_PENDING

You Have Uncommitted Work Pending

A DML operation was performed in the same transaction before an HTTP callout was attempted. Salesforce requires that all pending DML be committed or rolled back before issuing an external callout.

error
System.CalloutException: You have uncommitted work pending. Please commit or rollback before calling out.

What does this error mean?

Salesforce enforces a strict ordering rule within a synchronous transaction: you cannot mix DML and callouts in the same transaction in the DML-first order. The moment any DML statement (insert, update, delete, upsert, Database.insert(), etc.) executes, an open database transaction begins. If your code then tries to issue an HTTP callout, Salesforce throws this error because the database lock from the uncommitted transaction could be held for an unbounded time while waiting for the external service to respond.

DML (insert / update)
→
Http.send() 💥
→
CalloutException

Callouts before DML in the same transaction are permitted. The rule only applies when DML has already occurred prior to the callout in the same transaction context.

Common Causes

1. DML before a callout in the same method

The most direct case: an Apex method inserts or updates records then calls an external API to notify a third-party system about those records.

Apex — Bad Example 
// ❌ BAD: DML before callout in same transaction
public static void createAndNotify(Account acc) {
    insert acc; // DML — transaction begins

    HttpRequest req = new HttpRequest();
    req.setEndpoint('https://api.example.com/accounts');
    req.setMethod('POST');
    req.setBody(JSON.serialize(acc));
    new Http().send(req); // 💥 CalloutException
}

2. DML inside a trigger, callout in a class called by the same trigger

Trigger logic — or an automation like a Flow — performs DML. That same transaction then reaches an Apex action or invocable method that attempts a callout.

3. Savepoint creation before the callout

Database.setSavepoint() also counts as uncommitted work. Setting a savepoint and then attempting a callout triggers the same error.

How to Fix It

Solution 1: Move the callout before the DML

Reorder your logic so the external callout happens first. Use the response to populate field values, then perform the DML. This is valid because no database transaction is open yet when the callout is made.

Apex — Correct 
// ✅ GOOD: callout first, DML after
public static void createAndNotify(Account acc) {
    // Callout BEFORE any DML
    HttpRequest req = new HttpRequest();
    req.setEndpoint('https://api.example.com/accounts/reserve');
    req.setMethod('POST');
    HttpResponse res = new Http().send(req);

    if (res.getStatusCode() == 200) {
        Map<String,Object> body =
            (Map<String,Object>) JSON.deserializeUntyped(res.getBody());
        acc.External_Id__c = (String) body.get('externalId');
        insert acc; // DML safely after callout
    }
}

Solution 2: Delegate the callout to an async context

When you cannot reorder (e.g., you need the record Id for the callout payload), perform the DML first, then enqueue a Queueable job with Database.AllowsCallouts to make the callout after the transaction commits.

Apex 
// Step 1: DML first, then enqueue
public static void createAndNotify(Account acc) {
    insert acc; // DML — get the new Id
    System.enqueueJob(new AccountNotifyJob(acc.Id));
}

// Step 2: Queueable with callout runs after transaction commits
public class AccountNotifyJob
        implements Queueable, Database.AllowsCallouts {

    private Id accountId;

    public AccountNotifyJob(Id accId) {
        this.accountId = accId;
    }

    public void execute(QueueableContext ctx) {
        Account acc = [SELECT Id, Name FROM Account
                        WHERE Id = :accountId];
        HttpRequest req = new HttpRequest();
        req.setEndpoint('callout:ExternalAPI/accounts');
        req.setMethod('POST');
        req.setBody(JSON.serialize(acc));
        new Http().send(req);
    }
}
💡
Pro Tip: Named Credentials (callout:MyNamedCredential) are best practice for the callout endpoint. They handle OAuth token management and store credentials securely in Setup — no hardcoded URLs or secrets in your Apex code. Always use Named Credentials for any outbound integration in production orgs.
