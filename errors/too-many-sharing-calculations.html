Too Many Sharing Calculations

A DML operation triggered more sharing rule recalculations than Salesforce allows in a single transaction. This typically occurs when changing ownership on a large number of records at once in an org with complex sharing rule configurations.

error
System.LimitException: Too many sharing rule calculations. Perform fewer row updates per transaction.

What does this error mean?

Salesforce maintains record-level sharing entries in the AccountShare, OpportunityShare, and similar sharing tables. When you change a record's OwnerId or any field that a sharing rule evaluates, the platform must recalculate sharing for every affected record. If a single transaction triggers too many sharing rule evaluations — typically because hundreds or thousands of records are affected — this limit exception fires.

The limit is not a fixed public number but depends on the complexity of your org's sharing rules, criteria-based sharing, and ownership-based sharing configurations. Orgs with many sharing rules hit this limit at smaller record counts than orgs with simple sharing models.

Common Causes

1. Bulk OwnerId change in a single DML call

Reassigning ownership of hundreds or thousands of records in one update call is the most common trigger. Every record ownership change requires re-evaluation of all ownership-based and criteria-based sharing rules.

Apex — Bad Example
// ❌ BAD: OwnerId change on thousands of records in one transaction
List<Account> accounts = [
    SELECT Id, OwnerId FROM Account
    WHERE Region__c = 'EMEA'
];
for (Account a : accounts) {
    a.OwnerId = newOwnerId;
}
update accounts; // 💥 triggers sharing recalculation for all rows

2. Field change that triggers criteria-based sharing rules

Updating a field that is used as a criterion in a sharing rule (e.g., changing Account.Industry for thousands of accounts when a sharing rule grants access based on Industry) forces the platform to recalculate sharing for all affected accounts.

3. Territory reassignment on high-volume objects

Moving many accounts between territories in one transaction triggers both territory membership recalculation and the associated sharing recalculations for the opportunity team and case team access models.

How to Fix It

Solution 1: Use Batch Apex with a small batch size

Process ownership changes in small batches (50–100 records per execute). Each batch runs in its own transaction, limiting the sharing calculations per transaction to a manageable number.

Apex — Correct: Batch Apex
// ✅ Use Batch Apex to spread sharing recalculations across transactions
public class ReassignOwnerBatch implements Database.Batchable<SObject> {
    private Id newOwnerId;
    public ReassignOwnerBatch(Id ownerId) { this.newOwnerId = ownerId; }

    public Database.QueryLocator start(Database.BatchableContext bc) {
        return Database.getQueryLocator(
            'SELECT Id, OwnerId FROM Account WHERE Region__c = 'EMEA''
        );
    }

    public void execute(Database.BatchableContext bc,
                         List<Account> scope) {
        for (Account a : scope) { a.OwnerId = newOwnerId; }
        // Sharing calculations scoped to 200 records per batch
        update scope;
    }
    public void finish(Database.BatchableContext bc) {}
}
Database.executeBatch(new ReassignOwnerBatch(newOwnerId), 100);

Solution 2: Review and simplify sharing rules

Audit your org's sharing rules in Setup → Sharing Settings. Consolidate overlapping ownership-based rules, convert criteria-based rules to owner-based where possible, and delete any unused rules. Fewer rules mean fewer calculations per record change.

💡
Pro Tip: Use the Salesforce Optimizer (Setup → Optimizer) to identify sharing rule complexity. Orgs that hit this limit frequently should also consider using Manual Sharing, Apex Managed Sharing, or Platform Cache to supplement complex sharing models rather than relying on criteria-based sharing rules that fire on every bulk update.
