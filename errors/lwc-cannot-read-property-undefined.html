LWC: Cannot Read Properties of Undefined

A Lightning Web Component attempted to access a property on an undefined or null value — most often because wired Apex data or @api property data hasn't loaded yet when the getter or template expression runs.

error
TypeError: Cannot read properties of undefined (reading 'Name') — thrown in LWC component JavaScript

What does this error mean?

In LWC, wired data from Apex or the UI API is asynchronous. When a component renders for the first time, wired properties like @wire(getAccount) account contain an object where both account.data and account.error are undefined until the server responds. If a getter or template expression accesses a property on account.data before the wire resolves, JavaScript throws a TypeError that surfaces as a broken component in the UI.

Common Causes

1. Accessing nested wire data without null checks

The most common pattern: a getter returns this.account.data.Name without checking whether account.data is populated — it isn't on first render.

JavaScript
// ❌ BAD: accessing nested property before data arrives
import { LightningElement, wire } from 'lwc';
import getAccount from '@salesforce/apex/AccountController.getAccount';

export default class AccountDetail extends LightningElement {
    @wire(getAccount, { accountId: '$recordId' })
    account;

    get accountName() {
        // account.data is undefined until wire resolves
        return this.account.data.Name; // 💥 TypeError
    }
}

2. @api property not yet set by parent component

A child component's getter runs before the parent sets the @api property value. If the getter dereferences sub-properties of the @api value without a guard, it throws on first render.

3. connectedCallback accessing DOM elements before render

Calling this.template.querySelector() in connectedCallback() returns null because the component hasn't rendered yet. Calling a method on the null result throws TypeError.

How to Fix It

Solution 1: Use optional chaining and nullish coalescing

Replace all direct property access on wired data with optional chaining (?.) and provide fallback values with the nullish coalescing operator (??).

JavaScript — Correct
// ✅ Guard every property access on wired data
import { LightningElement, wire, api } from 'lwc';
import getAccount from '@salesforce/apex/AccountController.getAccount';

export default class AccountDetail extends LightningElement {
    @api recordId;

    @wire(getAccount, { accountId: '$recordId' })
    account;

    get accountName() {
        // Optional chaining + nullish coalescing
        return this.account?.data?.Name ?? 'Loading...';
    }

    get hasError() {
        return !!this.account?.error;
    }

    get isLoading() {
        return !this.account?.data && !this.account?.error;
    }
}

Solution 2: Guard rendering with conditional template directives

Use if:true (or lwc:if in API version 59+) to prevent template sections from rendering until data is available, eliminating the race condition at the template level.

💡
Pro Tip: Adopt a consistent three-state pattern for every wired call: isLoading (no data, no error), hasError (error present), and hasData (data present). Build a reusable wireState utility that derives these three flags from any wire result object — this eliminates the class of null-access errors entirely across your component library.
