SObject Types Do Not Match

You attempted to perform DML on a list containing a mix of different SObject types, or cast an SObject to an incompatible type. Salesforce DML operations require all records in a single call to be of the same SObject type.

error
System.SObjectException: SObject row was retrieved via SOQL without querying the requested field: Contact.AccountId — or — System.TypeException: Invalid conversion from runtime type Account to Contact

What does this error mean?

Salesforce's DML engine processes all records in a single insert, update, upsert, or delete call as a homogeneous batch of one SObject type. Mixing types in a List<SObject> and passing it to DML causes this type-mismatch error. The same error class also surfaces when you cast an SObject variable to the wrong concrete type.

Common Causes

1. Building a mixed-type DML list in a generic utility method

A generic helper accumulates records from multiple sources into a single List<SObject> and then passes the whole list to insert. This pattern fails unless the list is partitioned by type first.

Apex — Bad Example
// ❌ BAD: mixed SObject types in one list
List<SObject> records = new List<SObject>();
records.add(new Account(Name='Acme'));
records.add(new Contact(LastName='Smith'));

// Fails at DML — can't insert mixed types together
insert records; // 💥 SObject row was retrieved via SOQL without querying...

2. Incorrect cast of a polymorphic field result

Querying a polymorphic field like Task.What and casting the result to Opportunity when it actually points to a Case record throws a TypeException at the cast site.

3. Mixing standard and custom objects in factory/unit-of-work patterns

Enterprise patterns like the Unit of Work accumulate records across SObject types. Calling a single DML operation on the entire collection without type-bucketing causes this error.

How to Fix It

Solution 1: Partition records into typed lists before DML

Use instanceof to route each record to its own typed list, then perform separate DML calls per type.

Apex — Correct
// ✅ Keep separate lists per SObject type
List<Account> accounts = new List<Account>();
List<Contact> contacts = new List<Contact>();

for (SObject rec : mixedRecords) {
    if (rec instanceof Account) {
        accounts.add((Account) rec);
    } else if (rec instanceof Contact) {
        contacts.add((Contact) rec);
    }
}

if (!accounts.isEmpty()) insert accounts;
if (!contacts.isEmpty()) insert contacts;

Solution 2: Group by SObjectType using a Map

For generic code that must handle an open-ended set of SObject types, use SObject.getSObjectType() as the map key to group records, then iterate the map and DML each homogeneous group.

Apex — Type-Safe Map Pattern
// ✅ Group by type using Schema.SObjectType as key
Map<Schema.SObjectType, List<SObject>> byType =
    new Map<Schema.SObjectType, List<SObject>>();

for (SObject rec : allRecords) {
    Schema.SObjectType t = rec.getSObjectType();
    if (!byType.containsKey(t))
        byType.put(t, new List<SObject>());
    byType.get(t).add(rec);
}

for (List<SObject> group : byType.values()) {
    insert group; // each group is homogeneous
}
💡
Pro Tip: When writing generic utility methods that accept List<SObject>, always document whether the caller is responsible for type homogeneity or the method handles it internally. Adding an assertion System.assert(records.getSObjectType() != null) at the top of a DML utility is a lightweight guard that catches type mismatches in test runs before they reach production.
