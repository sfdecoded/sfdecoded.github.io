Batch Apex Scope Too Large

The batch size passed to Database.executeBatch() exceeds the maximum allowed value of 2,000 records per batch execution chunk.

error
System.AsyncException: Database.executeBatch called with a scope greater than 2,000. Specified scope: [N]

What does this error mean?

The second argument to Database.executeBatch(batchInstance, scope) specifies how many records are passed to each execute() invocation. Salesforce caps this value at 2,000. Passing any value greater than 2,000 throws a System.AsyncException immediately — the batch job is never submitted to the queue.

200
Default scope (no arg)
2,000
Maximum scope
1
Minimum scope
5M+
Records a single batch job can process

Note that a larger scope is not always better — each execute() transaction has its own governor limits, and a scope of 2,000 with a complex trigger chain can easily hit the DML row or CPU time limit.

Common Causes

1. Hard-coded scope value above 2,000

A developer configures the scope size via a custom setting or constant with a value of 2,500, 5,000, or higher, not realising the platform hard cap is 2,000.

Apex — Bad Example
// ❌ BAD: scope of 2001 exceeds the maximum
Database.executeBatch(new MyBatch(), 2001);
// Throws: Database.executeBatch called with a scope
// greater than 2000. Specified scope: 2001

2. Scope read from a custom setting that was changed post-deployment

A batch job reads its scope size from a custom setting (Batch_Config__c.Scope_Size__c). An admin updates the setting to 2,500 to "improve performance," causing the batch to fail immediately at submission time.

3. Dynamic scope calculation producing values over 2,000

A formula like totalRecords / numberOfBatches can produce values exceeding 2,000 if the total record count is large and the number of batches is small.

How to Fix It

Solution 1: Keep scope within the 2,000 limit

Set the scope to 2,000 or below. For most DML-heavy jobs, 200 is the safest default, providing ample headroom for trigger-fired child record operations.

Apex — Correct
// ✅ Use 200 for DML-heavy jobs (safe margin)
Database.executeBatch(new MyBatch(), 200);

// ✅ Use up to 2000 only for read-only or callout-heavy jobs
//    where DML row count isn't a concern
Database.executeBatch(new MyReadOnlyBatch(), 2000);

Solution 2: Right-size scope based on job type

Choose scope based on what your execute() method actually does — specifically whether it issues DML, makes callouts, or is read-only. Use the following heuristics as a starting point.

Apex — Right-Sizing by Job Type
// Choosing the right batch size:

// DML-heavy (triggers fire on child records)
// → Use 50–100 to stay under DML row limit
Database.executeBatch(new AccountOwnerBatch(), 50);

// Simple field updates, no complex triggers
// → Default 200 is safe and efficient
Database.executeBatch(new StatusUpdateBatch(), 200);

// Read-only aggregation, no DML
// → Maximise to 2000 for fewer job executions
Database.executeBatch(new ReportingBatch(), 2000);

// External callouts (one per record)
// → Use 1–10; callout limit is 100 per transaction
Database.executeBatch(new IntegrationBatch(), 10);
💡
Pro Tip: Expose batch scope as a Batch_Settings__mdt custom metadata record rather than a hard-coded value or a regular custom setting. Custom metadata is included in deployments and change sets, making scope configuration a code-reviewed, version-controlled decision rather than an admin-editable runtime value that can silently break batch jobs.
