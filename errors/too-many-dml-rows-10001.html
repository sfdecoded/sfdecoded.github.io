Too Many DML Rows: 10001

Your Apex transaction attempted to insert, update, delete, or upsert more than 10,000 records in a single synchronous transaction, hitting the DML row governor limit.

error
System.LimitException: Too many DML rows: 10001

What does this error mean?

Salesforce caps the total number of records that can be written in a single transaction. This limit is shared across all DML operations in the transaction — inserts, updates, deletes, upserts, and undeletes all count against the same pool. Once the cumulative row count reaches 10,001, the platform throws a System.LimitException and rolls back the entire transaction.

Importantly, Batch Apex does not automatically raise this limit — each execute() call is its own transaction, with its own 10,000 row cap. Batch Apex helps by splitting large datasets into smaller transactions, not by providing a higher per-transaction limit.

Common Causes

1. Processing large datasets in a single synchronous transaction

A scheduled job, a screen flow's bulk action, or a direct Apex invocation attempts to update tens of thousands of records at once — for example, a mass re-assignment of Account owners when a sales rep leaves.

2. Cascade DML: one record triggers updates to many children

A trigger on a parent object updates the parent record and then iterates over all child records to update them too. With a high-volume parent (e.g., an Account with thousands of Contacts or Opportunities), the combined row count of parent + children exceeds 10,000 in a single trigger transaction.

3. Multiple DML operations on overlapping datasets

A single transaction performs an insert of 5,000 records, then an update of 3,000 records, then a delete of 2,500 records. Each operation is individually within limits, but the cumulative total of 10,500 rows exceeds the cap.

How to Fix It

Solution 1: Use Batch Apex to chunk the work

Batch Apex is the primary tool for processing more than 10,000 records. Each execute() call processes a configurable chunk (default 200 rows, max 2,000), with its own fresh transaction and DML row counter.

Apex 
public class MassReassignBatch implements Database.Batchable<SObject> {

    private Id newOwnerId;

    public MassReassignBatch(Id ownerId) {
        this.newOwnerId = ownerId;
    }

    public Database.QueryLocator start(Database.BatchableContext bc) {
        return Database.getQueryLocator(
            'SELECT Id, OwnerId FROM Account WHERE OwnerId = :oldOwner'
        );
    }

    public void execute(Database.BatchableContext bc,
                         List<Account> scope) {
        for (Account acc : scope) {
            acc.OwnerId = newOwnerId;
        }
        // Processes 200 rows per transaction — well within limits
        Database.update(scope, false); // partial success
    }

    public void finish(Database.BatchableContext bc) {
        // Notify on completion
    }
}

// Kick off the batch — processes all records, 200 at a time
Database.executeBatch(new MassReassignBatch(newOwnerId), 200);

Solution 2: Check limits proactively with Limits class

Use Limits.getDmlRows() and Limits.getLimitDmlRows() to monitor how many DML rows have been consumed at key checkpoints and stop or reroute processing before hitting the ceiling.

Apex 
public static void processBatch(List<Account> accounts) {
    Integer remaining = Limits.getLimitDmlRows() - Limits.getDmlRows();
    System.debug('DML rows remaining: ' + remaining);

    if (accounts.size() > remaining) {
        // Too many records for this transaction — enqueue overflow
        List<Account> safe = new List<Account>(
            accounts.subList(0, remaining - 1)
        );
        update safe;
        // Queue the rest asynchronously
        System.enqueueJob(new AccountUpdateJob(
            accounts.subList(remaining - 1, accounts.size())
        ));
    } else {
        update accounts;
    }
}

Solution 3: Reduce unnecessary DML with conditional updates

Before performing DML, compare field values to determine which records actually need updating. Skipping records whose values haven't changed can dramatically reduce your row count.

Apex 
List<Account> toUpdate = new List<Account>();
Map<Id,Account> oldMap = Trigger.oldMap;

for (Account acc : Trigger.new) {
    // Only add records where the relevant field actually changed
    if (acc.Rating != oldMap.get(acc.Id).Rating) {
        toUpdate.add(acc);
    }
}

if (!toUpdate.isEmpty()) {
    update toUpdate; // Only update what changed
}
💡
Pro Tip: Use Limits.getDmlRows() in debug logs or in a System.debug() statement to measure DML row consumption at critical points in your transaction. Add this check in your trigger handlers to get early warning before you hit the limit in production. For large-scale data operations, always default to Batch Apex with a batch size of 200, which gives you comfortable headroom for any trigger-fired child updates.
