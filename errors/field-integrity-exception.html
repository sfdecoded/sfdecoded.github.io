FIELD_INTEGRITY_EXCEPTION

A field value violates one of Salesforce's field-level integrity rules — most commonly an invalid picklist value, a value assigned to a dependent picklist whose controlling field was not set, or a value that breaks a reference constraint.

error
FIELD_INTEGRITY_EXCEPTION: StageName: bad value for restricted picklist field: Pending Approval

What does this error mean?

FIELD_INTEGRITY_EXCEPTION is a DML-layer validation error that fires when a field value breaks the platform's definition of that field. The three most common triggers are: assigning a string to a restricted picklist that isn't an active picklist entry, populating a dependent picklist value without first setting the controlling picklist, or setting a lookup field to an Id that doesn't satisfy the field's filter criteria.

Common Causes

1. Restricted picklist value not in the active list

When a picklist is marked Restricted in Setup, only active picklist values are accepted. Assigning a string that was once valid but has since been deactivated, or a value from a different object or process, throws this error.

Apex — Bad Example
// ❌ BAD: assigning a value that violates picklist constraint
Opportunity opp = new Opportunity(
    Name      = 'Big Deal',
    StageName = 'Pending Approval', // not a valid picklist value
    CloseDate = Date.today().addDays(30)
);
insert opp; // 💥 FIELD_INTEGRITY_EXCEPTION

2. Dependent picklist value without the controlling field

Setting a dependent picklist (e.g., Case.Reason) without first setting its controlling field (e.g., Case.Type) causes an integrity failure because the platform cannot validate the dependency without the parent value.

3. Invalid record type assignment

Setting a RecordTypeId that is inactive, from a different object, or not assigned to the running user's profile triggers a field integrity exception during DML.

How to Fix It

Solution 1: Validate picklist values against Schema at runtime

Use Schema.DescribeFieldResult.getPicklistValues() to fetch active entries and validate inbound values before DML, particularly in integration code that receives field values from external systems.

Apex — Validate Before DML
// ✅ Validate picklist value at runtime using Schema
public static Boolean isValidPicklist(
        Schema.SObjectType objType,
        String fieldName,
        String value) {

    Schema.DescribeFieldResult dfr =
        objType.getDescribe()
               .fields
               .getMap()
               .get(fieldName)
               .getDescribe();

    for (Schema.PicklistEntry ple : dfr.getPicklistValues()) {
        if (ple.isActive() && ple.getValue() == value) {
            return true;
        }
    }
    return false;
}

// Usage
if (!isValidPicklist(Opportunity.getSObjectType(),
                    'StageName', stageName)) {
    throw new IllegalArgumentException(
        'Invalid StageName: ' + stageName);
}

Solution 2: Always set controlling fields before dependent fields

When building records that use dependent picklists, set the controlling field value before assigning the dependent field. The order of field assignment on the SObject literal matters.

Apex — Dependency Picklist Check
// ✅ For dependent picklists — verify parent value is set first
Case c = new Case();
c.Type    = 'Feature Request'; // controlling field
c.Reason = 'New Problem';      // dependent field

// Ensure controlling field is set BEFORE dependent field
// and that the dependent value is valid for the controlling value
insert c;
💡
Pro Tip: When processing inbound data from external systems (CSVs, API payloads, Heroku Connect), build a validation layer that maps external values to Salesforce picklist entries using a custom metadata type or a map populated from Schema. This isolates all picklist vocabulary translation in one place and prevents FIELD_INTEGRITY_EXCEPTION from reaching your DML layer.
