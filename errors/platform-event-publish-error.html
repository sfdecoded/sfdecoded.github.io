Platform Event Publish Error

One or more Platform Events failed to publish. EventBus.publish() returns per-event success/failure results that must be explicitly checked — unlike DML, failures do not automatically throw an exception.

error
EventBus.publish() SaveResult: isSuccess=false — LIMIT_EXCEEDED: Platform event publish limit reached, or FIELD_CUSTOM_VALIDATION_EXCEPTION on event fields.

What does this error mean?

Platform Events are published via EventBus.publish(), which returns a List<Database.SaveResult>. Unlike regular DML, publish errors do not throw exceptions — they are silently recorded in the result object. Developers who ignore the return value never see the failure, leading to missed events and hard-to-debug integration gaps.

The daily Platform Event publish limit is shared across the entire org, including all Apex transactions, Flows, and API calls.

Common Causes

1. Ignoring the SaveResult from EventBus.publish()

The most common cause of silent publish failures is simply not capturing or checking the return value of EventBus.publish().

Apex — Bad Example
// ❌ BAD: ignoring publish results — errors are silent
Order_Update__e evt = new Order_Update__e(
    Order_Id__c = order.Id,
    Status__c   = 'Shipped'
);
EventBus.publish(evt); // result ignored — failures swallowed

2. Daily publish limit exceeded

High-volume orgs processing thousands of transactions can exhaust the daily Platform Event allocation. Production orgs default to 250,000 publishes per day (250,000,000 for high-volume events with add-on licensing).

3. Required fields missing on event SObject

Platform Event objects can have required fields configured in Setup. Publishing an event without populating those fields returns a REQUIRED_FIELD_MISSING error in the SaveResult.

How to Fix It

Solution 1: Always inspect the SaveResult list

Capture the return value of EventBus.publish() and iterate through each result, logging or handling any failures. This is the minimum required for production-quality event publishing.

Apex — Correct
// ✅ Check Database.SaveResult from EventBus.publish()
List<Order_Update__e> events = new List<Order_Update__e>();
for (Order__c o : orders) {
    events.add(new Order_Update__e(
        Order_Id__c = o.Id,
        Status__c   = o.Status__c
    ));
}

List<Database.SaveResult> results = EventBus.publish(events);

for (Integer i = 0; i < results.size(); i++) {
    Database.SaveResult sr = results[i];
    if (!sr.isSuccess()) {
        for (Database.Error err : sr.getErrors()) {
            System.debug(LoggingLevel.ERROR,
                'Event publish failed [' + i + ']: ' +
                err.getStatusCode() + ' — ' + err.getMessage());
        }
    }
}

Solution 2: Implement a retry job for transient failures

For critical events, collect failed publishes and enqueue a Queueable retry. Persist persistent failures to a custom log object for monitoring dashboards.

Apex — Retry with Queueable
// ✅ Collect failed events and retry asynchronously
public class EventRetryJob implements Queueable {
    private List<Order_Update__e> failedEvents;

    public EventRetryJob(List<Order_Update__e> events) {
        this.failedEvents = events;
    }

    public void execute(QueueableContext ctx) {
        List<Database.SaveResult> retryResults =
            EventBus.publish(failedEvents);
        // Log persistent failures to a custom error log object
        for (Database.SaveResult sr : retryResults) {
            if (!sr.isSuccess()) {
                // insert Publish_Error__c record for monitoring
            }
        }
    }
}
