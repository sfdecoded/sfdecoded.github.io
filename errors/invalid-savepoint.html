Invalid Savepoint

You attempted to use a Savepoint that has already been rolled back, or you called Database.rollback() on a savepoint that was set before a callout in the same transaction.

error
System.TypeException: Invalid savepoint — the savepoint has already been rolled back or is otherwise invalid.

What does this error mean?

A Savepoint in Apex marks a point in a transaction to which you can roll back. Once Database.rollback(sp) is called, that savepoint is consumed and cannot be reused. Any subsequent call to Database.rollback(sp) with the same variable throws this exception.

Additionally, savepoints are invalidated if a callout is made after the savepoint is set — because an HTTP response from an external system cannot be undone by a database rollback.

Common Causes

1. Rolling back the same savepoint in multiple catch blocks

A single savepoint is captured before a try block, then referenced in multiple catch blocks. The second rollback call uses an already-consumed savepoint.

Apex — Bad Example
// ❌ BAD: rollback used twice on the same savepoint
Savepoint sp = Database.setSavepoint();
try {
    insert accounts;
    insert contacts;
} catch (Exception e1) {
    Database.rollback(sp); // first rollback — fine
}
try {
    insert opportunities;
} catch (Exception e2) {
    Database.rollback(sp); // 💥 savepoint already consumed
}

2. Making a callout after setting a savepoint

Code sets a savepoint, then issues an HTTP callout (Http.send()). After the callout returns, calling Database.rollback(sp) throws because the callout invalidated the savepoint.

3. Savepoint set in one method, rollback attempted in another after re-entry

A savepoint is captured in an outer method and passed to a utility class. If that utility class itself triggers DML that fires a trigger that rolls back to a different point, the outer savepoint may be invalidated.

How to Fix It

Solution 1: Create a new savepoint for each rollback scope

Each discrete unit of work that needs independent rollback capability should have its own savepoint. Set a new savepoint immediately before the work it is guarding.

Apex — Correct
// ✅ Create a new savepoint for each discrete rollback scope
Savepoint sp1 = Database.setSavepoint();
try {
    insert accounts;
    insert contacts;
} catch (Exception e) {
    Database.rollback(sp1);
    throw e; // re-throw after rollback if needed
}

Savepoint sp2 = Database.setSavepoint(); // fresh savepoint
try {
    insert opportunities;
} catch (Exception e) {
    Database.rollback(sp2); // rolls back to sp2 only
}

Solution 2: Use a single savepoint for all-or-nothing transactions

When the entire transaction must succeed or roll back atomically, use one savepoint that wraps all DML. If anything fails, roll back once to the single savepoint.

Apex — Single Savepoint Pattern
// ✅ Wrap all-or-nothing work in ONE savepoint scope
Savepoint sp = Database.setSavepoint();
try {
    insert accounts;
    insert contacts;
    insert opportunities;
    // All succeed — savepoint never consumed
} catch (Exception e) {
    Database.rollback(sp); // rolls back everything atomically
    System.debug('Transaction rolled back: ' + e.getMessage());
}
💡
Pro Tip: Savepoints count against the DML statement governor limit — each Database.setSavepoint() call consumes one DML statement slot. Avoid creating savepoints inside loops. The maximum number of savepoints per transaction is 5 in a synchronous context, so use them sparingly and only for genuinely complex multi-object atomic writes.
