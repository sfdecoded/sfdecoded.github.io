Future Method Cannot Be Called From Future/Batch Context

You called an @future method from another @future method, a Batch Apex execute(), or a Queueable job — which Salesforce does not allow for @future methods.

error
System.AsyncException: Future method cannot be called from a future or batch method.

What does this error mean?

The @future annotation marks a method to run asynchronously after the current transaction completes. However, Salesforce prohibits chaining @future calls — you cannot call a @future method from within another @future method, from a Batch Apex execute() method, or from a Batch finish() method.

This restriction prevents unbounded async chains that could cascade across the platform. The rule is: @future is a dead-end context — once you're in it, you cannot spawn more @future work.

Calling ContextCan call @future?Can enqueue Queueable?
Synchronous Apex / Trigger✓ Yes✓ Yes
@future method✗ No✓ Yes
Batch execute()✗ No✓ Yes
Batch finish()✗ No✓ Yes
Queueable execute()✗ No✓ Yes (chain)
Scheduled Apex execute()✗ No✓ Yes

Common Causes

1. Shared service class called from multiple contexts

A utility service class contains a method that calls @future. That service is called both from a trigger (fine) and from a Batch Apex execute() method (throws). The same code path works in one context and fails in another.

2. @future method triggering a DML that fires a trigger

An @future method inserts or updates a record. A trigger on that object calls another @future method. Because the trigger still runs in the future context, the nested @future call throws.

3. Batch finish() spawning more async work via @future

A Batch Apex finish() method is commonly used to kick off follow-on work, but calling @future there is not permitted. This surprises developers who expect finish() to behave like a synchronous context.

How to Fix It

Solution 1: Replace @future with Queueable

Queueable jobs can be enqueued from virtually any async context, including @future methods (with one enqueue per transaction limit from future context) and Batch execute(). Migrate your logic from @future to a Queueable class.

Apex — Before (Broken) 
// ❌ BAD: @future calling @future (via trigger)
public class NotificationService {
    @future
    public static void sendEmail(Set<Id> recordIds) {
        // ... does DML that fires a trigger ...
        // That trigger also calls @future — BOOM
    }
}
Apex — After (Correct) 
// ✅ GOOD: Queueable can be enqueued from async contexts
public class NotificationJob implements Queueable {
    private Set<Id> recordIds;

    public NotificationJob(Set<Id> ids) {
        this.recordIds = ids;
    }

    public void execute(QueueableContext ctx) {
        // safe async context, full 60s CPU, callouts with
        // Database.AllowsCallouts interface
        for (Id recId : recordIds) {
            // process each record
        }
    }
}

// Enqueue from anywhere — trigger, batch, @future, etc.
System.enqueueJob(new NotificationJob(recordIds));

Solution 2: Guard with System.isFuture() / System.isBatch()

If refactoring immediately is not possible, add a runtime context check. Skip the @future call when already in an async context and handle that path differently (e.g., process inline or skip).

Apex 
public static void scheduleWork(Set<Id> ids) {
    if (System.isFuture() || System.isBatch() || System.isQueueable()) {
        // already async — enqueue a Queueable instead
        System.enqueueJob(new NotificationJob(ids));
    } else {
        // synchronous context — @future is safe
        LegacyService.processAsync(ids);
    }
}
💡
Pro Tip: Use System.isFuture(), System.isBatch(), System.isQueueable(), and System.isScheduled() to detect your execution context at runtime. Building a context-aware service layer that routes to the appropriate async mechanism based on these flags is a clean architectural pattern that prevents this class of errors entirely.
