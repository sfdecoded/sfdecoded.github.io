List Has More Than 1 Row for Assignment to SObject

A SOQL query assigned to a single SObject variable returned more than one record. Salesforce only allows this assignment when the query produces exactly one row.

error
System.QueryException: List has more than 1 row for assignment to SObject

What does this error mean?

When you assign a SOQL query result directly to an SObject variable (rather than a List), the Apex runtime enforces a strict contract: the query must return exactly one row. Zero rows throws List has no rows; two or more rows throws this error.

The exception is a runtime error — the compiler does not catch it. Code that works perfectly in a developer sandbox (where only one matching record exists) can fail silently in production as data grows.

⚠️
Data-growth trap: This error is frequently absent in development and staging environments and first appears in production after data accumulates. Always prefer the List query pattern for any query that does not guarantee a unique result.

Common Causes

1. Querying by a non-unique field

Filtering by Name, Industry, Status, or any other non-unique field and assigning directly to an SObject — works until a second matching record is created.

Apex — Bad Example
// ❌ BAD: SObject assignment when multiple rows exist
Account acc = [SELECT Id, Name FROM Account
                  WHERE Industry = 'Technology'];
// Throws if more than one Account matches

2. Missing WHERE clause filters

A broad query intended to fetch "the one" record of a given type — for example, a configuration custom object where only one record is expected — starts failing once an admin inadvertently creates a second record.

3. Polymorphic lookups returning multiple types

Queries on polymorphic relationship fields like What on Task or Who on Event can match records across multiple SObject types, increasing the likelihood of returning multiple rows.

How to Fix It

Solution 1: Add LIMIT 1 and use a List

The safest universal fix: always query into a List<SObject> and add LIMIT 1 when you need only one result. Then check isEmpty() before accessing index zero.

Apex — Correct
// ✅ Option A: enforce uniqueness with LIMIT 1
Account acc = [SELECT Id, Name FROM Account
                  WHERE External_Id__c = :extId
                  LIMIT 1];

// ✅ Option B: query into a List and handle multiple results
List<Account> accs = [SELECT Id, Name FROM Account
                       WHERE Industry = 'Technology'];
if (accs.isEmpty()) { return; }
Account acc = accs[0]; // take first, or iterate all

Solution 2: Full null-safe single-record pattern

The recommended production-grade pattern combines LIMIT 1, a List assignment, and an empty-check — handling both the "zero rows" and "multiple rows" failure modes simultaneously.

Apex — With Null Safety
// ✅ Full safe pattern — no ListException, no NullPointerException
List<Account> results = [
    SELECT Id, Name, OwnerId
    FROM   Account
    WHERE  External_Id__c = :externalId
    LIMIT  1
];
if (results.isEmpty()) {
    System.debug('No account found for: ' + externalId);
    return;
}
Account acc = results[0];
💡
Pro Tip: Reserve the direct SObject assignment syntax (Account a = [SELECT ...]) only for queries that filter on a guaranteed-unique indexed field such as Id or a field with a unique constraint — and only when you want an exception to be thrown on zero results. In all other cases, query into a List.
