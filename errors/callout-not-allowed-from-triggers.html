Callout Not Allowed From Triggers

You attempted to make an HTTP callout or web service call directly inside an Apex trigger, which Salesforce prohibits in synchronous trigger context.

error
System.CalloutException: Callout from triggers are currently not supported.

What does this error mean?

Salesforce enforces a strict rule: synchronous Apex triggers cannot make HTTP callouts or web service invocations (REST, SOAP, or any HttpRequest call). This restriction exists because triggers run inside a database transaction — if the external service were slow or unreachable, the transaction would stall, consuming a database lock for an indeterminate period and risking timeout cascades across your org.

The error surfaces immediately when your trigger — or any class synchronously called by your trigger — executes an Http.send(), a stub generated from a WSDL, or an @HttpCallout-annotated mock in tests without the proper annotation setup.

Importantly, this restriction also applies to code paths called indirectly: if your trigger calls a utility class that calls a service class that issues a callout, the exception still fires even though the callout is not in the trigger file itself.

Common Causes

1. Direct Http callout inside a trigger body

The most direct cause — an HttpRequest / Http.send() block placed inside a trigger file or a helper method called synchronously from one.

Apex — Bad Example 
// ❌ BAD: direct callout inside trigger
trigger OpportunityTrigger on Opportunity (after insert) {
    for (Opportunity opp : Trigger.new) {
        HttpRequest req = new HttpRequest();
        req.setEndpoint('https://api.example.com/notify');
        req.setMethod('POST');
        new Http().send(req); // 💥 throws CalloutException
    }
}

2. Callout buried inside a service class called from a trigger

A trigger delegates to a handler or service class, which in turn calls an integration class. The callout restriction applies to the entire synchronous call chain, not just the trigger file.

3. Test class missing @isTest(SeeAllData=false) or mock setup

In unit tests, calling a method that issues an HTTP request without a registered HttpCalloutMock or WebServiceMock throws this same exception. The test framework blocks real network calls.

How to Fix It

Solution 1: Move the callout to an @future method

Annotate a static method with @future(callout=true). The trigger enqueues the work; the callout executes asynchronously after the transaction commits, where callouts are fully supported.

Apex — Correct 
// ✅ GOOD: delegate to @future(callout=true)
trigger OpportunityTrigger on Opportunity (after insert) {
    Set<Id> oppIds = new Set<Id>();
    for (Opportunity opp : Trigger.new) {
        oppIds.add(opp.Id);
    }
    OpportunityCalloutService.notifyExternalSystem(oppIds);
}

public class OpportunityCalloutService {
    @future(callout=true)
    public static void notifyExternalSystem(Set<Id> oppIds) {
        for (Id oppId : oppIds) {
            HttpRequest req = new HttpRequest();
            req.setEndpoint('callout:ExternalAPI/notify');
            req.setMethod('POST');
            req.setBody('{"id":"' + oppId + '"}');
            HttpResponse res = new Http().send(req);
        }
    }
}

Solution 2: Use Queueable Apex with callout support

For more complex scenarios — passing rich objects, chaining jobs, or needing better error handling — implement Queueable with the Database.AllowsCallouts marker interface. This gives you a full class context, instance variables, and the ability to chain subsequent jobs.

Apex 
public class OpportunityNotifyJob
        implements Queueable, Database.AllowsCallouts {

    private List<Opportunity> opps;

    public OpportunityNotifyJob(List<Opportunity> opps) {
        this.opps = opps;
    }

    public void execute(QueueableContext ctx) {
        for (Opportunity opp : opps) {
            HttpRequest req = new HttpRequest();
            req.setEndpoint('callout:ExternalAPI/notify');
            req.setMethod('POST');
            req.setBody(JSON.serialize(opp));
            HttpResponse res = new Http().send(req);
            // handle res.getStatusCode() etc.
        }
    }
}

// In trigger:
System.enqueueJob(new OpportunityNotifyJob(Trigger.new));

Solution 3: Register a mock in tests

In unit tests, implement HttpCalloutMock and register it with Test.setMock() before calling the method under test. Without this, even a correctly structured async callout class will throw the exception when exercised synchronously in tests.

Apex — Test Class 
@isTest
private class OpportunityCalloutServiceTest {

    private class MockCallout implements HttpCalloutMock {
        public HttpResponse respond(HttpRequest req) {
            HttpResponse res = new HttpResponse();
            res.setStatusCode(200);
            res.setBody('{"status":"ok"}');
            return res;
        }
    }

    @isTest
    static void testNotify() {
        Test.setMock(HttpCalloutMock.class, new MockCallout());
        Test.startTest();
        OpportunityCalloutService.notifyExternalSystem(
            new Set<Id>{UserInfo.getUserId()}
        );
        Test.stopTest();
    }
}
💡
Pro Tip: Prefer Queueable over @future for new callout work — Queueable classes support instance variables, proper error handling via AsyncException, and job chaining. Reserve @future(callout=true) only for simple, fire-and-forget notifications where you're passing only primitive or Set<Id> parameters.
